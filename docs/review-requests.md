# Review Requests

TrustLoop professionals collect reviews by sharing **unique, time-limited links** with clients. This phase implements review request creation, token validation, dashboard management, and automatic completion when a client submits a review.

> Email and SMS delivery are **not** implemented in this phase. Professionals copy links manually.

## Why requesting reviews is essential

Reviews are TrustLoop's core asset, but they only exist if clients take action. Most clients won't spontaneously visit a profile — professionals need a **low-friction way to ask**. Review requests bridge that gap: one link, one client, one review.

## Why review requests are separate from reviews

| Concern | Review request | Review |
|---------|----------------|--------|
| Purpose | Invitation / tracking | Published feedback |
| Actor | Professional creates | Client submits |
| Lifecycle | pending → completed/expired | Permanent once created |
| Identity | Tied to client email | Public name + private email |

Separating them keeps the `reviews` table clean (only real feedback) while `review_requests` tracks outreach state, expiration, and shareable tokens without polluting review data.

## How unique links improve UX

Each request gets a **UUID token** — unguessable, single-use in practice (completed after submission). Benefits:

- **No account required** — clients open the link and submit
- **Email locked** — prevents token sharing with wrong person (`EMAIL_MISMATCH` if email differs)
- **Expiration** — 30-day default reduces stale links
- **Verified reviews** — reviews from a valid request get `verified: true`

Example URL: `https://yoursite.com/review/request/{token}`

```
        │
        ▼

```

## Token generation

- Tokens are **UUID v4** generated by Postgres (`gen_random_uuid()` default on `review_requests.token`)
- Unique index on `token` prevents collisions
- `expires_at` = `created_at` + `REVIEW_REQUEST.TOKEN_EXPIRY_DAYS` (30)

Tokens cannot be forged: clients must possess the exact UUID from the database.

## Validation

### `getRequestByToken(token)`

1. Validate UUID format (Zod)
2. Load request + profile join
3. If `pending` and past `expires_at` → auto-expire, return `EXPIRED`
4. If `expired` → `EXPIRED`
5. If `completed` → `ALREADY_COMPLETED`

### Review submission with token

1. `getRequestByToken` must succeed
2. `reviewerEmail` must match `request.email` (case-insensitive)
3. Review inserted with `review_request_id` and `verified: true`
4. `complete_review_request` RPC marks request completed

## Completion flow

`complete_review_request(p_token)` is a **SECURITY DEFINER** RPC because anonymous clients submit reviews without auth to update `review_requests`.

The RPC:

1. Locks the row by token
2. Rejects if not found, expired, or already expired status
3. Sets `status = 'completed'`, `completed_at = now()`
4. Idempotent if already completed

Called from `createReviewAction` after successful review insert.

## RLS policies

| Table | Policy |
|-------|--------|
| `review_requests` SELECT | Public (token is the secret capability) |
| `review_requests` INSERT | Profile owner only |
| `review_requests` UPDATE | Profile owner (expire); RPC for complete |

## Future email integration

When email is added:

1. `createRequest` triggers send with `getReviewRequestUrl(token)`
2. No schema changes required — `email` column already stores recipient
3. Optional: track `sent_at`, resend via existing `resendRequest()`
4. Dashboard "Copy link" remains as fallback
